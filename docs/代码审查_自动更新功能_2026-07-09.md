# 代码审查报告：自动更新数据功能

> 审查对象：`scripts/crawl.py`、`scripts/model.py`、`scripts/daily_update.bat`、`.ps1`、`.github/workflows/daily-update.yml`、`index.html`
> 审查时间：2026-07-09
> 结论先行：**爬虫本身能跑、能连、逻辑基本正确；"看不到更新"的真正原因不在爬虫算法，而在「触发机制没接上」+「失败被静默吞掉」+「模型结果不稳定」+「前端不轮询」。** 下面逐一说明。

## 一、总体印象

先把你最关心的问题直接回答掉：

| 你问的 | 结论 |
|---|---|
| ① 数据更新的触发机制是否正确配置？ | ❌ **没有真正自动触发**。本地 `.bat`/`.ps1` 只是手动脚本，仓库里没有任何定时任务注册；GitHub Actions 只更新云端仓库、不会回写你本地 `D:\lottery-ai` 的文件。 |

## 二、🔴 必须修的问题（Blocker）

### B1. 自动更新从来没有被"定时"触发过

**位置**：`scripts/daily_update.bat`、`scripts/daily_update.ps1`（仓库根目录无 `schtasks` / 任务计划 XML）

**现象**：这两个脚本写好了，但仓库里**没有任何"注册成定时任务"的配置或说明**。也就是说，在你自己电脑上，没有任何东西会每天自动去跑爬虫。数据永远是"你上次手动双击 bat"的状态。

**叠加问题**：GitHub Actions（`daily-update.yml`）只把更新提交到**云端仓库**，并不会动你本地 `D:\lottery-ai\data\` 的文件。两套机制是割裂的——云端在跑、本地不动。

**后果**：新开奖后，你打开本地网页看到的是旧数据 → 你以为"自动更新坏了"。

**修复方案（二选一，别两套都当真）：**

- **方案 A：本地定时（推荐你这种单机使用）**
  用任务计划程序注册一个每天定时任务，指向 `daily_update.bat`：
  ```bat
  schtasks /create /tn "LotteryAutoUpdate" /tr "D:\lottery-ai\scripts\daily_update.bat" /sc daily /st 23:30
  ```
  或在「任务计划程序」图形界面里手动建，触发器设"每天 23:30"，操作指向 `daily_update.bat`。

- **方案 B：云端定时（多人/在线访问）**
  确保仓库已 `git push` 到 GitHub、已开启 Pages；Actions 每天定时跑，用户通过 `https://xxx.github.io/lottery-ai/` 访问最新结果。

### B2. 爬虫失败被"静默吞掉"，你永远分不清"没数据"还是"挂了"

**位置**：`scripts/crawl.py` `main()` 第 250–266 行（两个 `try/except` 把异常全吞掉）；第 284–285 行 `main()` **永远 `return 0`**。

**为什么是 bug**：
```python
try:
    new = crawl_ssq(session)
    total_new['ssq'] = new
except Exception as e:
    print(f'[SSQ] 爬取失败: {e}')   # 只打印，然后 ssq_new=[]
    total_new['ssq'] = []
...
return 0   # ← 无论成功失败，退出码都是 0（成功）
```
- 调度器 / 日志看到"退出码 0 = 成功"，不会报警。
- `.crawl_meta.json` 里 `ssq_new: 0` 既可能是"今晚确实没开奖"，也可能是"网站把爬虫拦了 / 网络超时 / 接口改版报错"。**你无法区分。**

**修复方案**：让失败"显形"——
```python
def main():
    status = {}
    try:
        rate_limit(); new = crawl_ssq(session)
        status['ssq'] = {'new': len(new), 'status': 'ok'}
    except Exception as e:
        status['ssq'] = {'new': 0, 'status': 'error', 'error': str(e)}
        print(f'[SSQ] 爬取失败: {e}')

    try:
        rate_limit(); new = crawl_dlt(session)
        status['dlt'] = {'new': len(new), 'status': 'ok'}
    except Exception as e:
        status['dlt'] = {'new': 0, 'status': 'error', 'error': str(e)}
        print(f'[DLT] 爬取失败: {e}')

    summary = {
        'ssq_new': status.get('ssq', {}).get('new', 0),
        'dlt_new': status.get('dlt', {}).get('new', 0),
        'ssq_status': status.get('ssq', {}).get('status'),
        'dlt_status': status.get('dlt', {}).get('status'),
        'timestamp': datetime.now(TZ).isoformat(),
    }
    with open(meta_file, 'w', encoding='utf-8') as f:
        json.dump(summary, f, ensure_ascii=False)

    had_error = any(s.get('status') == 'error' for s in status.values())
    return 1 if had_error else 0   # ← 有失败就返回非 0，调度器能告警
```
这样 `.crawl_meta.json` 会多一个 `ssq_status` 字段，一眼就能看出是"无数据"还是"出错了"。

### B3. 模型每次结果都不一样（非确定性）→ 工作流"无变更就跳过"永远失效

**位置**：`scripts/model.py` `mc_simulation()` 第 113–121 行，用 `np.random.choice(...)` 但**全程没有设置随机种子**。

**证据（我实跑两次，输入数据完全没变）**：
- 第一次 SSQ 红球 MC Top3：`24(100) / 30(94.27) / 25(80.51)`，蓝球：`3(100) / 2(85.73) / 1(83.77)`
- 第二次：`24(100) / 30(95.33) / 28(82.87)`，蓝球：`3(100) / 2(94.16) / 14(91.94)`

**为什么是 bug**：
- 在 GitHub Actions 里，`git diff --quiet data/` **永远非空**（因为 MC 分数每次都变）→ 每天都会 `commit + push` 一次"daily update"，哪怕没有任何新开奖 → 刷屏 + 浪费 Actions 额度。设计文档第 5 步"无变更则跳过"形同虚设。
- 模型不可复现，别人没法复现你的结果。

**修复方案**：固定随机种子（与 RF 的 `random_state=42` 一致）：
```python
def mc_simulation(periods, ball_pool, is_front=True, n_sim=10000, seed=42):
    rng = np.random.default_rng(seed)          # 可复现
    ...
    chosen = rng.choice(ball_pool, size=pick_count, replace=False, p=weights)
    ...
```
（`np.random.default_rng(seed).choice` 支持 `p=`，行为和原来一致。）

## 三、🟡 应该修的问题（Suggestion）

### S4. `.crawl_meta.json` 写了却没人读（机制脱节 / 死代码）

**位置**：`crawl.py` 第 268 行注释写"写标记文件供 model.py 判断是否有新数据"，但 `model.py` **从头到尾没读这个文件**；workflow 里 `model.py` 是无条件执行的。

**后果**：设计意图"仅 crawl 有新增时才重算模型"没有实现，每天白跑模型（且因 B3 必然产生变更）。

**修复**：要么让 `model.py` 读取 meta 判断后跳过；要么接受"每天重算"但必须先修 B3 让结果稳定。别留一个没人用的文件。

### S5. 本地脚本把 Python 路径写死在 F 盘

**位置**：`daily_update.bat` 第 4–5 行、`daily_update.ps1` 第 4 行：`F:\马维斯AI\Marvis\...\python.exe`

**风险**：这个"马维斯"运行时一旦移动 / 卸载 / 升级，脚本立刻静默失败；再叠加 B2（退出码恒为 0），你根本发现不了。

**修复**：
- 优先用 PATH 里的 `python`；
- 或用一个固定虚拟环境（如 `D:\lottery-ai\.venv\Scripts\python.exe`）；
- 在脚本开头检测路径是否存在，不存在就打印明确错误并 `exit 1`。

---

### S6. 前端只加载一次数据，没有任何轮询 / 自动刷新

**位置**：`index.html` 第 10 行 `DOMContentLoaded → loadAllData()`，之后再无 `setInterval` / `visibilitychange` / 定时 `fetch`。

**后果（对应你问的"定时器/事件监听/轮询"）**：**这套机制根本不存在**。即使爬虫成功更新了 `data/*.json`，只要页面还开着，用户看到的就一直是旧数据，必须手动刷新。

**额外提醒**：`index.html` 用 `fetch('./data/...')`，如果直接双击用 `file://` 打开，浏览器会因同源策略/CORS **拒绝加载数据**（页面会一直卡在骨架屏或报错）。必须用本地 HTTP 服务器打开，例如：
```bat
cd D:\lottery-ai && python -m http.server 8000
```
然后访问 `http://localhost:8000`。

**可选修复（看你是否要"开着页面也能看到最新数据"）**：加一个轮询，例如每 5–10 分钟重新 `loadAllData()`（需做好并发与错误兜底）。至少把"必须用 HTTP 访问"写进文档。

## 四、💭 小改进（Nit）

### N7. 工作流 cron 与设计文档不一致
`daily-update.yml` 第 5 行 `cron: '0 15 * * *'`（UTC 15:00 = 北京时间 23:00）；设计方案文档写的是"北京时间凌晨 2:00"（`0 18 * * *`）。功能上 23:00 能覆盖当晚开奖，但和文档对不上容易困惑，建议统一并加时区注释。

### N8. 大乐透抓取范围写死 `26001–26999`
`crawl.py` 第 168–169 行。2026 年内没问题；到 2027 年（期号 ≥27001）后会悄悄拉不到新数据（且不报错，只显示"无新数据"）。建议改成"取最近 N 期"或动态计算起止。

### N9. 双色球接口健壮性
`cwl.gov.cn` 的 `findDrawNotice` 有时需要 `x-requested-with: XMLHttpRequest` 或先访问首页拿 cookie，否则可能被拦。当前 `get_headers()`（第 44–52 行）没带这些。本次测试能拿到数据，但夜间调度时若被拦，结合 B2 会被静默吞掉。建议补上必要请求头，并把失败显式记录。
