# AI 推荐模块 · 技术可行性评估报告

> 评估日期：2026-07-09
> 评估对象：`scripts/model.py`（算法）、`index.html`（推荐展示与合并）、`data/*_model.json`（模型产出）、`docs/AI彩票分析工具_设计方案计划书.md`（原始规划）
> 评估视角：当前架构是否支撑后续升级迭代，重点针对 AI 推荐模块

---

## 0. 执行摘要（先说结论）

**架构层面：支撑增量升级，但有明确边界。** 当前系统是"纯静态站点 + 静态 JSON + 浏览器端 JS"，云端靠 GitHub Pages 部署、靠 GitHub Actions 每日跑脚本。这种架构做**数据分析、可视化、规则推荐**完全够用，但**任何需要"记忆/存档/训练服务"的能力**（历史推荐记录、回测、深度学习训练）都必须在不引入后端的前提下用"文件 + Actions 提交"的方式实现，否则就要推翻现有部署模型。

**AI 模块层面：有优化空间，但必须认清一个物理事实——**
彩票每期开奖近似独立随机事件，**没有任何模型能稳定预测中奖号码**。本项目设计书第 40 行也已写明定位是「AI 辅助选号分析工具，非中奖保证系统」。因此本报告的"优化"目标应定义为：**提升分析维度、增强可解释性、补上验证机制**，而不是"提高中奖率"。

**最关键的缺口（决定后续一切）：** 系统**从不保存历史推荐记录**，导致你问的"中奖概率偏差率验证"目前根本无法做——没有历史预测 vs 实际开奖的对照数据。好消息是：这个能力可以在现有静态架构内用"JSON 日志 + Actions 每日提交"补上，不需要后端。

## 1. 现状盘点（基于真实代码）

### 1.1 架构现状
| 维度 | 现状 | 来源 |
|---|---|---|
| 部署形态 | 纯静态站点（GitHub Pages），无后端、无数据库 | `daily-update.yml` 仅打包 `index.html`+`data/` |
| AI 算法位置 | 服务端脚本 `model.py` 在 Actions 里跑，产出 JSON；前端 `index.html` 读取并展示 | `model.py` / `index.html` |
| 前端合并逻辑 | `refreshRecommend()`（index.html:1304）按蒙特卡洛频率加权随机采样生成 4 组 | `index.html:1311-1396` |
| 数据规模 | 双色球约 500 期、大乐透约 503 期 | `data/*_data.json` |

### 1.2 现有算法（`model.py`）真实构成
- **三套子模型**：随机森林 `RandomForestClassifier(n_estimators=200)`、朴素贝叶斯 `GaussianNB`、蒙特卡洛加权采样（`n_sim=10000`，已固定 `seed=42` 保证确定性）。
- **特征工程**（`build_features`，model.py:24）：仅 3 类——遗漏值、近 5/10/20 期频率、平均间隔。
- **建模范式**：把每个号码当成**独立的二分类问题**（"这个号码下期会不会出"），再取 Top15。
- **产出**：每个号码组（红/蓝、前/后）各输出 rf/nb/mc 三个 Top15 评分列表。

### 1.3 必须直说的两个问题（影响"概率准确性"判断）
1. **展示的"评分/星级"不是中奖概率。** `index.html:1365` 的公式是：
   `score = min(round(avgRate/maxRate*65 + 20 + random()*10), 95)`
   即 70% 来自 MC 频率、20 分保底、**30% 是纯随机噪声**。星级由这个分数推出。这给用户一种"系统算出了中奖概率"的错觉，实际上没有统计含义。
2. **"推荐理由"是写死的套话，不是模型推导。** `index.html:1395` 写死："基于蒙特卡洛 10000 次模拟加权随机采样重新组合……每次重新生成结果不同，仅供参考比较"。设计书规划的"每组的生成依据说明"（计划书:64）并未真正落地。

### 1.4 设计意图 vs 实际实现的落差（重要）
| 计划书规划 | 实际代码 | 落差 |
|---|---|---|
| 每次推荐结果存档 `model_results`（计划书:97，用于回测） | 无存档逻辑 | ❌ 未实现 → 无法做偏差率验证 |
| 三模型取交集→共识号码（权重×2），非交集取排名均值（计划书:426-427） | `refreshRecommend()` 只用 MC 频率做随机采样，RF/NB 产出几乎未参与最终推荐 | ❌ 未实现 |
| 训练集最近 500 期、留最后 50 期验证（计划书:392） | 用全量训练，**无验证集、无准确率评估** | ❌ 未实现 |
| 回补概率 ≈ 1-e^(-λ·遗漏值)（计划书:309）等规则特征 | 未体现 | ❌ 未实现 |

## 2. 四大方向深度评估

### 2.1 现有推荐算法的优化空间

**① 模型参数调优（低成本，可做）**
- RF：`n_estimators=200` 可网格搜索到 300~500；`max_depth`/`min_samples_leaf` 未限制，易过拟合小样本，建议加约束。
- MC：`n_sim=10000` 足够，但采样权重仅用"近 50 期频率"，可叠加遗漏/冷热因子。
- **预期收益**：对"分析维度丰富度"有边际提升，对真实命中率几乎无影响（见 2.2 根因）。

**② 特征工程改进（中低成本，推荐做）**
- 现有 3 类特征偏薄。可增补：冷热度（hot/cold）、连号概率、和值分布、AC 值、区间分布、奇偶比 —— 这些恰好是计划书:309-374 已规划但未落地的规则特征。
- **架构兼容性**：纯前端/脚本内计算，不需要后端，可直接在 `build_features` 扩展。**完全契合现有静态架构。**

**③ 训练数据扩充（低成本，推荐做）**
- 当前 ~500 期（约 3 年）。可回溯到中彩网/500.com 全量历史（双色球自 2003 年、数千期）。
- 注意：数据增多主要改善**统计稳定性**，不会突破"随机事件"天花板。

**④ 范式层面的硬伤（必须如实告知）**
当前"每个号码独立二分类再取 TopN"的范式，对"6 个号码一组不放回抽取"的彩票来说在统计上是不严谨的。更合理的范式是**学习排序（Learning to Rank）/ 直接对组合打分**，但代价显著上升，且收益仍受随机事件天花板限制。

### 2.2 中奖概率的统计准确性验证方法

**这是你问的重点，也是当前最大的能力缺口。**

**为何现在做不了：** 系统从不记录"某天推荐了什么 + 实际开了什么"，没有对照数据，自然算不出偏差率。

**可行的验证方案（契合静态架构）：**
1. **补建"预测存档"**：在每日 Actions 跑完 `model.py` 后，把当天的 Top-N 推荐（红/蓝、前/后）连同目标期号写入 `data/predict_history.json`（追加，由 Actions 提交）。—— 这正好补上计划书:97 的规划，且不引入后端。
2. **开奖后回写结果**：次日运行得知实际开奖后，把命中情况（命中几个红/蓝）回写到对应记录。
3. **偏差率指标**：
   - **理论基线**：随机选 6 红的中奖概率 = 1/C(33,6) ≈ 1/110 万；至少中 1 红的概率 ≈ 1 - C(27,6)/C(33,6) ≈ 66.4%。**任何模型的命中率必须和这个基线比，才知道有没有"超越随机"的边缘。**
   - **实测指标**：推荐组合"至少中 k 红"的实际频率 vs 理论频率；Top1 号码命中率 vs 单球理论命中率（6/33≈18.2%）。
4. **诚实结论预期**：基于随机事件性质，模型的实测命中率**应无限接近理论基线**。若显著偏离，反而说明采样/数据有 bug，而非"模型更准"。

**关键提醒**：验证的目的应是"证明系统没有虚假宣传"，而非"证明 AI 更准"。当前前端把随机噪声包装成"评分/概率"的做法，正是验证要纠正的对象。

### 2.3 引入更先进模型（深度学习 / 集成学习）

**集成学习（Stacking / XGBoost）—— 中成本，推荐优先级高于深度学习**
- 在现有 RF+NB 基础上加 XGBoost/LightGBM，再做一层元学习器融合。实现仍在 `model.py` 内（scikit-learn 生态），**无需后端、不破坏静态架构**。
- 预期：分析维度更稳，但同样受随机事件天花板限制。

**深度学习（LSTM / Transformer 序列化建模）—— 高成本，低预期收益，建议谨慎**
- 技术上可把"每期开奖序列"当成时间序列用 LSTM/Transformer 建模。但：
  - **物理限制**：彩票序列近似白噪声，深度学习无法从无规律中提取规律，预期命中率仍≈理论基线。
  - **架构冲突**：训练需要算力/时长，与"GitHub Actions 免费 runner + 静态 Pages"模式不匹配；若坚持，只能在 Actions 内训练并把权重/产出 JSON 随站点发布（推理仍在浏览器端），不能做在线训练。
  - **成本**：数据预处理、序列建模、调参、过拟合防控，工作量远大于收益。
- **建议**：除非目标是"用先进模型做产品卖点/演示"，否则**不推荐**为提升准确性引入深度学习。

### 2.4 推荐结果可解释性增强

**现状**：展示理由是写死的套话（index.html:1395），RF/NB 的模型内部信号完全没用上。

**真实可解释方案（契合静态架构，推荐做）：**
1. **RF 特征重要性导出**：`model.py` 训练后可取 `rf.feature_importances_`，按"遗漏值/近5期频率/近10期频率/近20期频率/平均间隔"归一化，写入 `data/*_model.json`。前端即可展示"某号码被推荐，主要因为：遗漏值偏高（均值回归）+ 近10期频率上升"。
2. **SHAP 值**（更严谨）：scikit-learn 可算 TreeExplainer，给出每个号码的逐特征贡献。计算量小，可在 Actions 内完成。
3. **规则化说明**：复用计划书:309-374 的冷热/和值/奇偶/AC 值规则，对推荐组合做"体检式"解读（如"本组奇偶比 3:3，和值处于历史 95% 区间内"）。
4. **把"评分"改名为"热度参考指数"**：消除"这是中奖概率"的误导，符合计划书"非中奖保证系统"的定位。

## 3. 技术风险评估

| 风险 | 等级 | 说明 | 缓解 |
|---|---|---|---|
| **统计误导风险** | 🔴 高 | 前端把随机噪声包装成"中奖概率/星级"，可能误导用户、引发合规/信任问题 | 重命名"概率"为"参考指数"；补回测验证；加免责声明 |
| **架构边界风险** | 🟡 中 | 纯静态 + Pages 部署，无法原生支撑"历史存档/在线训练/数据库" | 用"JSON 日志 + Actions 提交"兜底；确需后端时评估另起轻量 API |
| **数据质量风险** | 🟡 中 | 数据源从 cwl.gov.cn 切到 500.com（已修），若 500.com 改版/封 IP，抓取会断裂 | 已在 `crawl.py` 做单源失败隔离 + 黄色警告；建议保留备用源 |
| **小样本过拟合** | 🟡 中 | ~500 期训练 RF，特征少易过拟合 | 加 `max_depth` 约束、交叉验证、扩充历史数据 |
| **期望值管理风险** | 🔴 高 | 用户/受众可能误以为"AI 能预测中奖" | 产品定位、文案、UI 全程强调"辅助分析、非预测保证" |
| **维护风险** | 💭 低 | 算法逻辑分散在 `model.py` + `index.html` 两处，合并逻辑在前端且含随机性 | 把合并/打分逻辑收敛到 `model.py` 产出，前端只渲染 |

## 4. 预期收益（按方向）

| 改造项 | 预期收益 | 不做的代价 |
|---|---|---|
| 预测存档 + 回测（P0） | 让"概率准确性"可验证、可对外交代；填补计划书规划缺口 | 永远无法证明/反驳"是否靠谱"，信任风险持续 |
| 评分改名 + 免责（P0） | 消除误导，规避合规风险 | 持续误导用户，潜在投诉/舆情 |
| 特征工程扩充（P1） | 分析维度更丰富，产品更专业 | 分析面板单薄，竞争力弱 |
| RF 特征重要性导出（P1） | 真实可解释，替代套话 | "推荐理由"形同虚设 |
| 集成学习 Stacking（P2） | 模型更稳，技术叙事更完整 | 错过中等成本的能力提升 |
| 深度学习（P3） | 仅"先进"标签价值 | 高成本 + 低准确性收益，不推荐为 accuracy 做 |

## 5. 实施优先级建议

> 原则（贴合 PM 角色）：先补"验证与诚实"底座，再做"分析厚度"，最后才考虑"模型炫技"。不堆砌、不镀金。

**P0（必须先做，否则后续都无法验证）**
1. 新增 `data/predict_history.json` 预测存档 + Actions 每日追加/回写（补计划书:97）。
2. 前端"评分/星级"改名为"热度参考指数"，加显著免责声明；修正 index.html:1365 的随机噪声公式。

**P1（性价比最高，纯静态架构内可完成）**
3. `model.py` 特征工程扩充（冷热/连号/和值/AC 值等计划书已规划特征）。
4. 导出 RF 特征重要性 / SHAP 到 `data/*_model.json`，前端渲染真实推荐理由。
5. 把"三模型合并"从前端随机采样收敛为 `model.py` 产出的确定性共识（落实计划书:426 的交集加权）。
6. 训练数据回溯至全量历史。

**P2（模型升级）**
7. 引入 XGBoost/LightGBM + Stacking 元学习器（仍在 `model.py` 内，不破坏架构）。
8. 建立留一/时序交叉验证，输出"模型 vs 理论基线"的命中率对照（闭环 2.2）。

**P3（谨慎，仅当需"先进"标签）**
9. 序列深度学习（LSTM/Transformer）——仅建议作为演示/技术叙事，不为提升准确性。

## 6. 一句话结论

当前架构**能支撑 AI 模块的持续迭代**，且大部分升级（特征、可解释性、集成、预测存档）都能在"静态站点 + GitHub Actions"框架内完成、不破坏现有云端部署。但有两件事必须优先：**先把"概率"变诚实、先补上预测存档做回测验证**——否则任何"模型更准"的说法都既无法证实也无法证伪。至于"用深度学习大幅提命中率"，请务必管理好预期：在近似随机的彩票面前，它是高成本、低收益的选择。
